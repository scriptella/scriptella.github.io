Introduction

Scriptella is a Java-based ETL and script execution tool. Its primary scripting language is ordinary SQL executed through JDBC. Other providers add non-JDBC data sources and scripting languages, allowing one ETL file to combine them.

A typical ETL process collects data from one or more sources and loads it into other sources, optionally transforming values along the way. Scriptella provides three basic elements:

Connection
A connection to a data source such as a database, directory service, or XML file.
Script
Code to execute: SQL, JavaScript, JEXL, or another language understood by the selected driver.
Query
Code that reads a data source. A query can contain nested queries and scripts.

Queries work like for each statements. They iterate through rows—such as JDBC result-set rows, LDAP entries, XML elements, or regular-expression matches—and expose each row's columns or attributes as variables to nested elements.

When to use Scriptella

Scriptella is a good fit when you need to:

  • Store and execute database SQL without depending on vendor command-line tools.
  • Work with several data sources—for example, reading statistics from one database and storing them in another.
  • Keep queries visible as SQL instead of expressing them through a visual query designer.
  • Generate test data during execution instead of storing large generated files.
  • Perform schema evolution or data migration.

For one-to-one database replication, a dedicated replication tool will usually be a better choice.

System requirements

Scriptella 1.3 requires a Java 8 JDK or JRE. Memory use depends on the ETL and the connection providers it loads; in-process databases may need additional heap.

Installation

  1. Download the binary distribution.
  2. Unpack it and add <SCRIPTELLA_DIR>/bin to your system PATH.
  3. Confirm Java is available by running java -version. You can use JAVA_HOME to select a JDK.
  4. Optionally put JDBC drivers needed by your ETL in <SCRIPTELLA_DIR>/lib, or set a connection's classpath attribute.

For Windows:

set PATH=%PATH%;SCRIPTELLA_DIR\bin

For Unix-like systems:

export PATH=${PATH}:SCRIPTELLA_DIR/bin

The binary distribution has this layout:

scriptella/
├── bin/          launch scripts
├── docs/
│   ├── api/      API documentation
│   └── dtd/      ETL DTD and element documentation
└── lib/          libraries loaded by the launch scripts

The main libraries are:

  • scriptella-core.jar — core execution classes.
  • scriptella-drivers.jar — built-in data-source drivers and adapters.
  • scriptella-tools.jar — command-line, Ant, and supporting tools.
  • scriptella.jar — the all-in-one Scriptella JAR. Driver-specific dependencies still need to be supplied separately.

Script syntax

For the complete list of XML elements and attributes, see the Scriptella DTD documentation.

This example shows the main ETL elements:

<!DOCTYPE etl SYSTEM "http://scriptella.org/dtd/etl.dtd">
<etl>
  <description>Copy bugs between databases</description>
  <properties>
    <include href="script.properties"/>
    driver=org.jdbcDriver
  </properties>

  <connection id="in" driver="$driver" url="$sourceUrl"/>
  <connection id="out" driver="$driver" url="$targetUrl"/>

  <script connection-id="out">
    <include href="dbschema.sql"/>
  </script>

  <query connection-id="in">
    SELECT * FROM Bug
    <script connection-id="out">
      INSERT INTO Bug VALUES (?ID, ?priority, ?summary, ?status)
    </script>
  </query>
</etl>

<etl> is the root element of every Scriptella ETL file.

<properties>

Defines properties that can be substituted in other ETL elements. If a property is declared more than once, the first value takes precedence. Use <include> to read properties from an external file.

Properties in this element can be overridden by system or external properties supplied by the execution environment, such as an Ant task.

<connection>

Defines a connection to a data source. Required attributes depend on the driver. JDBC connections commonly use url, user, and password.

id
Required when an ETL declares more than one connection. Scripts and queries refer to this value with connection-id.
url
A driver-specific URL. File-based drivers accept absolute locations or paths relative to the ETL file.
driver
A driver class name or alias. The default, auto, attempts to infer the driver from the URL. See the driver matrix.
classpath
Optional driver libraries, resolved relative to the ETL file. Separate multiple paths with the platform path separator.
lazy-init
When true, delays opening the connection until an element uses it.

Element text sets driver-specific connection properties. Libraries in Scriptella's boot classpath take precedence over libraries named in classpath.

<script>

Executes its text through the selected connection. A script may include external files, use variables from a parent query, and run conditionally with if or unless.

<script connection-id="out">
  INSERT INTO target(id, name) VALUES (?id, ?name)
</script>

<query>

Reads rows from its connection and executes nested elements once per row. Each column or attribute becomes a variable visible to nested queries and scripts.

<query connection-id="in">
  SELECT id, name FROM source
  <script connection-id="out">
    INSERT INTO target VALUES (?id, ?name)
  </script>
</query>

<onerror>

Handles a matching script or query error. Handlers can match error codes or message text and may retry execution. Keep matches narrow so unexpected failures still stop the ETL.

Expressions and variable substitution

Scriptella exposes properties and query-row values to nested elements. Use $name or ${name} in XML attributes and script text. JDBC scripts also support bind variables such as ?name and expressions such as ?{firstName + ' ' + lastName}.

INSERT INTO person(id, full_name)
VALUES (?id, ?{firstName + ' ' + lastName})

Bind variables are preferable to string substitution for SQL values because the JDBC driver handles their types and escaping.
